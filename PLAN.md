# rook — build plan

A Go-based code intelligence tool that ingests SCIP indexes, stores them in Postgres, caches in Redis, exposes them over GraphQL and MCP, and uses Claude to generate grounded plain-English walkthroughs of symbols.

The project exists for two reasons at once:

1. Dalton needs a concrete Go artifact in the `github.com/MapleRook` portfolio that maps directly to the Sourcegraph Code Understanding [IC2] job listing (Greenhouse req 625, posted 2026-04-10).
2. Dalton wants to re-read and re-internalize his own existing projects (claude-palace, AC7 Decoded, contractordraft) at a deeper level before interviews. `rook explain <symbol>` is the learning engine that forces him through his own code.

Every design decision in this doc serves one or both of those goals. If a decision doesn't, cut it.

## What rook does (user-facing)

```
$ rook index ~/code/claude-palace
indexing claude-palace...
  ran scip-python
  ingested 1,247 symbols, 4,891 occurrences across 83 files
  done in 6.2s

$ rook lookup palace_search
palace_search (function)
  defined at: src/palace/search.py:42
  signature:  def palace_search(query: str, wing: str = None, limit: int = 10) -> list[Memory]
  called by:  12 sites
  calls:      chroma_client.query, _rerank_by_recency, Memory.from_chunk

$ rook explain palace_search
palace_search runs a semantic query against ChromaDB, scoped optionally to a
single wing. It calls the Chroma client with the query embedding, pulls back
up to (limit * 3) raw hits, then passes them to _rerank_by_recency which
applies a time-decay factor. The final list is wrapped in Memory objects via
Memory.from_chunk. No error handling around the Chroma call because an empty
result set returns as [] rather than raising.
  (generated by claude-haiku-4-5, grounded on the definition + 4 call sites)

$ rook serve
rook listening on:
  graphql: http://localhost:4000/graphql
  mcp:     stdio

$ rook mcp  # registers the mcp server for Claude Code consumption
```

### Database schema (Phase 1)

```sql
CREATE TABLE repos (
    id           SERIAL PRIMARY KEY,
    name         TEXT NOT NULL UNIQUE,
    root_path    TEXT NOT NULL,
    indexed_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE documents (
    id           SERIAL PRIMARY KEY,
    repo_id      INTEGER NOT NULL REFERENCES repos(id) ON DELETE CASCADE,
    relative_path TEXT NOT NULL,
    language     TEXT NOT NULL,
    UNIQUE (repo_id, relative_path)
);

CREATE TABLE symbols (
    id                 SERIAL PRIMARY KEY,
    repo_id            INTEGER NOT NULL REFERENCES repos(id) ON DELETE CASCADE,
    scip_symbol        TEXT NOT NULL,
    display_name       TEXT NOT NULL,
    kind               TEXT NOT NULL,
    signature          TEXT,
    documentation      TEXT,
    UNIQUE (repo_id, scip_symbol)
);
CREATE INDEX idx_symbols_display_name ON symbols (display_name);

CREATE TABLE occurrences (
    id              SERIAL PRIMARY KEY,
    document_id     INTEGER NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    symbol_id       INTEGER NOT NULL REFERENCES symbols(id) ON DELETE CASCADE,
    start_line      INTEGER NOT NULL,
    start_col       INTEGER NOT NULL,
    end_line        INTEGER NOT NULL,
    end_col         INTEGER NOT NULL,
    role            TEXT NOT NULL  -- definition | reference | import | write
);
CREATE INDEX idx_occurrences_symbol_id ON occurrences (symbol_id);
CREATE INDEX idx_occurrences_document_id ON occurrences (document_id);

CREATE TABLE explanations (
    id          SERIAL PRIMARY KEY,
    symbol_id   INTEGER NOT NULL REFERENCES symbols(id) ON DELETE CASCADE,
    content     TEXT NOT NULL,
    model       TEXT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_explanations_symbol_id ON explanations (symbol_id);
```

Phase 7 (cross-repo) will add:

```sql
CREATE TABLE cross_repo_links (
    id              SERIAL PRIMARY KEY,
    from_symbol_id  INTEGER NOT NULL REFERENCES symbols(id),
    to_symbol_id    INTEGER NOT NULL REFERENCES symbols(id),
    link_kind       TEXT NOT NULL  -- import | call | type_ref
);
```

## Phases

Each phase produces a runnable artifact and ends with a commit tagged `phase-N`. Dalton may pause after any phase, use the tool for a week, then resume.

### Phase 1 — Ingest and store
**Goal.** Go CLI that runs scip-python against a repo and loads the resulting SCIP index into Postgres.

**Exit criteria.**
- [x] `rook index <path>` works against `~/code/claude-palace`
- [x] Postgres has populated `repos`, `documents`, `symbols`, `occurrences` tables
- [x] `psql` can `SELECT display_name FROM symbols WHERE repo_id = 1 LIMIT 20;` and see real function names
- [x] Unit tests for the SCIP protobuf decoder path
- [x] Integration test using testcontainers-go

**Teaching goals for Dalton.** Go project layout, cobra CLI patterns, pgx connection pooling, working with a real protobuf schema, shelling out to subprocesses safely.

**Notes.**
- scip-python is a separate binary Dalton will install. `rook index` shells out to it, writes the resulting `index.scip` to a temp file, parses it with the sourcegraph/scip Go lib, and writes to Postgres in a single transaction per repo.
- Handle large repos with a streaming parse, not a read-everything-into-memory parse.
- The ingest loop should be concurrent at the document level (worker pool), not the symbol level. Teaches concurrency without overcomplicating.

---

### Phase 2 — Query and cache
**Goal.** CLI subcommands that answer real questions from the Postgres index, with Redis caching.

**Exit criteria.**
- [ ] `rook lookup <name>` returns definition + signature + call count
- [ ] `rook refs <name>` lists every reference site (repo:file:line)
- [ ] `rook callers <name>` returns distinct symbols that reference the target
- [ ] `rook callees <name>` returns distinct symbols referenced inside the target's body
- [ ] Redis caches `lookup` and `callers` results for 5 minutes keyed on `(repo, symbol)`
- [ ] `rook --no-cache` bypasses Redis
- [ ] Integration tests cover both cache-hit and cache-miss paths

**Teaching goals.** Go interfaces for pluggable storage, caching patterns, understanding how SCIP occurrences encode callees (hint: they don't directly; we infer callees by looking at references inside a definition's line range).

---

### Phase 3 — The explain command
**Goal.** Take a symbol, assemble grounding context, ask Claude for a plain-English walkthrough, cache the result.

**Exit criteria.**
- [ ] `rook explain <symbol>` prints a grounded paragraph
- [ ] Grounding context = definition source, up to 8 nearest call-site snippets, signature, documentation
- [ ] Default model: claude-haiku-4-5-20251001
- [ ] `--deep` switches to claude-sonnet-4-6
- [ ] Explanations cached in the `explanations` table keyed by symbol_id + model
- [ ] `--refresh` regenerates and overwrites
- [ ] Prompt lives in `internal/explain/prompts.go` as a versioned template
- [ ] API key read from `ANTHROPIC_API_KEY` env var

**Teaching goals.** Prompt engineering with grounding, Anthropic SDK, context-window budgeting, deterministic prompt testing.

**Prompt shape (draft).**
```
You are a code reader. You will receive one symbol from a real codebase
along with its definition and a sample of call sites. Your job is to
explain in plain English what this symbol does, what it depends on, and
what might surprise someone reading it for the first time.

Rules:
- Ground every claim in the provided source. Do not speculate.
- If the call sites contradict the docstring, say so.
- 4 to 8 sentences. Stop when the picture is complete.
- No marketing words. No "leverages," "robust," "comprehensive."

SYMBOL
{name} ({kind})
signature: {signature}
doc: {doc}

DEFINITION
```{language}
{definition_source}
```

CALL SITES ({n} shown)
{call_sites}
```

The model is instructed against the same voice rules Dalton writes in. The output should read like Dalton wrote it.

### Phase 4 — GraphQL API
**Goal.** Expose the same queries over GraphQL via gqlgen.

**Exit criteria.**
- [ ] `rook serve` starts an HTTP server on :4000
- [ ] `schema.graphql` defines Repo, Document, Symbol, Occurrence, Explanation types
- [ ] Queries: `repo`, `symbol`, `references`, `callers`, `callees`, `explain`
- [ ] Resolvers delegate to the same `query` package the CLI uses. No duplicated logic.
- [ ] GraphQL playground available at `/graphql`
- [ ] Basic request logging middleware

**Teaching goals.** GraphQL schema design, gqlgen code generation workflow, resolver patterns, separating transport from business logic.

---

### Phase 5 — MCP server (the stopping point)
**Goal.** Wrap the query layer in an MCP server so Claude Code can ask rook questions about Dalton's own repos.

**Exit criteria.**
- [ ] `rook mcp` starts an MCP server over stdio
- [ ] MCP tools registered: `rook_lookup`, `rook_refs`, `rook_callers`, `rook_callees`, `rook_explain`
- [ ] Each tool has a clear description and a JSON schema for parameters
- [ ] Adding rook to Dalton's `claude_desktop_config.json` / Claude Code MCP config works end to end
- [ ] Asking Claude Code "how does palace_search work" in a fresh session causes Claude to call `rook_explain` and return a grounded answer
- [ ] README has a short "use with Claude Code" section with copy-pasteable config

**Teaching goals.** MCP protocol basics, tool schema design, stdio transport, integrating a custom tool into Claude Code's client.

**This is the Sourcegraph interview walk-off.** At the end of Phase 5, rook is a miniature Sourcegraph 7.0: it consumes SCIP, exposes code intelligence to AI agents over MCP, and demonstrates exactly the thesis in their 2026-02-25 blog post. Stop here, re-read this plan, run the tool on all three personal repos, and feel how much of the codebases return to memory.

### Phase 7 — Cross-repo mode (optional moonshot)
**Goal.** Index multiple repos into one rook instance and resolve cross-repo symbol links.

**Exit criteria.**
- [ ] `rook index` accepts multiple paths in a single invocation
- [ ] `cross_repo_links` table populated by matching external symbols from SCIP
- [ ] `rook refs <symbol> --cross-repo` finds references in other indexed repos
- [ ] GraphQL type `Symbol` grows a `crossRepoReferences` field
- [ ] README section explicitly references the 2026-01-21 Sourcegraph cross-repo navigation blog post as the design inspiration

**Teaching goals.** This is the exact problem shape the Code Understanding team works on. If Dalton reaches this phase before applying, the cover letter gets an entire paragraph about it.

## Out of scope (do not build)

- Authentication, RBAC, multi-user
- A custom indexer or tree-sitter parser
- Kubernetes manifests or Helm charts

- Telemetry, analytics, feature flags
- A Sentry / error-reporting integration
- Any form of "user accounts"

Each of these has a plausible reason to exist in a real product. None of them serve the two goals at the top of this doc.

## Development conventions

- **Commits.** Conventional commits. One logical change per commit. Never skip pre-commit hooks.
- **Tests.** Every new package gets at least one test before the phase ships. Integration tests use `testcontainers-go` so they run against real Postgres and real Redis, not mocks.
- **Lint.** `golangci-lint run` is green before each commit. Config lives in `.golangci.yml`.
- **Make targets.**
  - `make test` runs all tests
  - `make lint` runs golangci-lint
  - `make build` produces `./rook`
  - `make docker-up` / `make docker-down` manage compose
  - `make migrate` runs goose migrations against local Postgres
- **Comments in code.** Only when a reader would otherwise be surprised. No docstrings on obvious functions.
- **Error handling.** Real errors at boundaries. No `panic` in library code. No wrapping errors with the same message as the cause.
- **No AI-generated marketing copy.** All user-facing strings and README prose should read like Dalton wrote them. If the fresh agent finds itself typing "empowering developers to..." stop and rewrite.
